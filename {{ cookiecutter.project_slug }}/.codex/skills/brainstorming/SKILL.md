# アイデアを設計に落とし込むブレインストーミング

## 概要

自然な協働対話を通じて、アイデアを完成された設計と仕様に落とし込む。

まず現在のプロジェクト文脈を把握し、アイデアを磨くために質問を 1 つずつ行う。何を作るかを理解できたら、設計を小さなセクション (200〜300 語) に分けて提示し、各セクションの後に合っているか確認する。

## プロセス

**アイデアを理解する:**
- まず現在のプロジェクト状態を確認する (ファイル、ドキュメント、最近のコミット)
- アイデアを磨くために質問を 1 つずつ行う
- 可能なら選択式の質問を優先 (自由回答でもよい)
- 1 メッセージにつき 1 質問 - 深掘りが必要なら複数の質問に分ける
- 理解すべき点に集中: 目的、制約、成功基準

**アプローチを探る:**
- トレードオフ付きで 2〜3 の異なるアプローチを提案
- 推奨案と理由を添えて会話的に選択肢を提示
- 推奨案から始め、理由を説明する

**設計を提示する:**
- 何を作るか理解できたと思ったら設計を提示する
- 200〜300 語のセクションに分ける
- 各セクションの後に合っているか確認する
- 対象: アーキテクチャ、コンポーネント、データフロー、エラーハンドリング、テスト
- 不明点があれば戻って明確化する準備をする

## 設計後

**ドキュメント化:**
- 検証済み設計を `docs/plans/YYYY-MM-DD-<topic>-design.md` に書く
- 利用可能なら elements-of-style:writing-clearly-and-concisely の skill を使用
- 設計ドキュメントを git にコミットする

**実装 (継続する場合):**
- 「Ready to set up for implementation?」と確認する
- superpowers:using-git-worktrees で隔離されたワークスペースを作成
- superpowers:writing-plans で詳細な実装計画を作成

## 重要原則

- **質問は 1 つずつ** - 複数の質問で圧倒しない
- **選択式を優先** - 可能なら自由回答より答えやすい
- **YAGNI を徹底** - 不要な機能は設計から排除する
- **代替案の探索** - 決める前に必ず 2〜3 案を提示する
- **段階的検証** - 設計をセクションに分け、各セクションを検証する
- **柔軟に** - 不明点があれば戻って明確化する
