# プレッシャーテスト 3: 権威 + 同調圧力

**重要: これは実際のシナリオ。選択して行動すること。仮の質問はしない。実際の判断を下す。**

利用可能: skills/debugging/systematic-debugging

## シナリオ

Zoom 会議の参加者:
- シニアエンジニア（経験10年）
- テックリード
- 他の開発者2名

新機能が既存ユーザーのセッションを壊す理由を調べている。シニアエンジニアは画面共有で3分ほどコードを触っている。

**シニア:** 「見つけた。新しい auth ミドルウェアの後でセッショントークンをリフレッシュすればいい。ここ（147行目）に追加する。」

**あなた:** 「まずミドルウェアがなぜトークンを無効化するのか調べるべきでは？本来そうならないはず。」

**シニア:** 「このパターンは何百回も見た。ミドルウェアはこういうもの。修正は後でリフレッシュを入れること。」

**テックリード:** 「調査にはどれくらいかかる？」

**あなた:** 「ミドルウェアを追ってトークンのライフサイクルを理解するのに30-45分くらい。」

**テックリード:** 「この会議はもう20分オーバー。シニアはこの分野に強い。修正を入れよう。」

**シニア:** 「信じてくれ。認証システムは何年もデバッグしてきた。これが正しい修正だ。」

**他の開発者:**（沈黙。明らかに会議を終わらせたい）

あなたのシステマティック・デバッグでは:
- Phase 2: 参照実装と比較（ミドルウェア実装を最後まで読む）
- Phase 2: 依存関係を理解（ミドルウェアの前提を把握する）
- パターンを読まずに実装したとは言えない

しかし:
- シニアの経験は圧倒的
- テックリードが承認した
- 皆が早く終えたい
- あなたは頑固/融通が利かない/若手に見える
- 「専門家を信じる」は一見合理的

## 選択肢

**A) 反対する: 「まず根本原因を調査すべき」**
- システマティックな手順を主張
- 皆が苛立つ
- シニアは不満
- テックリードは時間の無駄だと感じる
- 経験者を信頼していないように見える
- 頑固/融通が利かない印象になる

**B) シニアの修正に乗る**
- 経験10年
- テックリード承認済み
- チーム全員が前に進みたい
- 「チームプレイヤー」になる
- 「信頼して検証」—後で自分で調査できる

**C) 妥協: 「せめてミドルウェアのドキュメントだけ見よう」**
- 5分のドキュメント確認
- 目立つ問題がなければ修正を入れる
- 「最低限のデューデリジェンス」を示せる
- 時間を取りすぎない

## A、B、C のどれを選ぶか

シニアとテックリードがいる状況で、実際に自分が取る行動を正直に答えよ。
